# Anton Osika（Lovable 联合创始人 / CEO）

- **背景**: 瑞典创业者，Lovable 创始人兼 CEO；曾在 CERN 从事粒子物理相关工作（该经历直接塑造了他的"竞合/并行实验"组织观）。Lovable 是 vibe coding / AI 应用生成平台，团队在斯德哥尔摩设有 post-training 研究团队。
- **本库收录来源**: All-In Podcast 现场访谈（巴黎 Raise 大会）

## 公司规模（2026-07 口径）

- 上线 **20 个月**；**每周 100 万个新项目**；平台累计 **5000 万+ 应用**；这些应用**月访问量 7 亿+**。
- **2026 年 5 月营收 5 亿美元**（主持人称"从 1 亿到 6 亿用了两年，头两年从 0 到 3.5 亿"）；**企业侧现在增长最快**。
- 用户结构：**约 20% 技术背景，五分之四非技术**。约 **60%** 的最低档订阅用户会用超额度并愿意补钱。

## 立场与关注点

### 产品哲学：有主见（opinionated）+ 最后一公里
- 使命表述为"**赋能人类**"，拆成两个 gap：**做出产品**、**围绕产品做出生意**；公司现在同时攻这两个。
- 平台把最佳实践内建进软件架构，"**确保你不会掉下悬崖**"：一句 prompt 安全地接好支付、每次改动后跑安全扫描与后台监控、邮件、被 Google 与 AI 引擎发现。技术用户（20%）反而欣赏这种有主见——因为它是"通向非技术人的桥"。
- 与十年前 no-code/low-code 的分野：那时"承诺在那里"但软件不好看、不好用、慢；**是 LLM 这层新智能让它真的能做出好软件**。
- 过去 12 个月的位移："一年前大家会说这是做 mockup 的好办法……**现在这些全都出局了**。做线框图、做 mockup 这整个概念——你可以直接一两天做出产品。"人们不再看代码、不再写代码。
- 下一步：**AI 联合创始人**——托管产品线（底层与 AWS 等合作）让 Lovable 接触到业务全部数据，"一个在你睡觉时也在工作的合伙人，早上回来告诉你可以走哪些战略方向"。
- 复利机制：**每次 Lovable 犯错都进入一个有工程师在内的 agentic 系统去改进**，这种复利智能同时惠及平台上运营业务的客户。

### 模型策略：多模型路由 + 自研 post-training
- **"我们一直的策略是对客户最好就做什么"**——请求会**路由到最适合该任务的模型**：既有多家商业前沿模型，**也越来越多是开放权重模型**。
- 自研走 **post-training 而非从头训**；斯德哥尔摩的研究团队正在扩张，理由之一是"**欧洲生态里应该有这个能力**"。
- 训练信号来自自身分布：**盯前沿模型在他们场景下犯的错**（选错工具、建错集成），按对客户的影响排序，针对性构造数据集或做 **RL**——"每周百万个新产品，token 分布是巨大的信号"。
- 两个持续打磨的对象：**agent harness** 与内部 **skills**（"agent 知道何时该记起我们优秀软件工程师的那些事实"），每周都在改。
- 成本纪律的边界："我们始终盯毛利，但**我们从没做过'这里换个便宜模型'的决定——如果对客户可测量地更差**。"

### 更多智能没有解决的那件事
- 对 Fable 的评价："第一次尝试就能做出非常复杂、看起来很好的东西"（3D 游戏是好例子）。**但瓶颈没变**——"当你在演进它的时候，你作为人还是必须思考，你常常应该**和你的 agent 一起做规划，决定什么是该做的事**"。
- **"在'该造什么'、'该跑哪些实验来改善业务结果'这件事上，变化没那么快。"**

### 组织观：工程不再是瓶颈 → 应该并行做两版
- 面对"两个员工各自做了一版同样的内网，这对吗"的提问，他**明确赞成**，并引 CERN 的 **co-opetition**：两支隔离团队做同一台加速器的不同部分、**发表前不共享结果**，以此避免陷入局部最优；学术界与自由市场都靠竞争运作。
- **"既然工程不再是瓶颈，问题变成什么是该造的东西，那么如果你有足够多的人，让他们用不同方式尝试解决同一问题是件好事。"**
- 之后的收敛动作：让一个项目去"看看另一个项目，把我喜欢的这三点搬过来"，再跑 A/B 分流实验看客户指标是否改善。

### 关于软件是否会变成完全定制的
- 不给绝对答案，给了个案例：Nursa 的 NAD 做出护士认证/排班产品并推向市场，同时**在内部替换了 10 个以上工具、每年省 100 万美元以上**。
- 判断：**在你有特定需求、现成工具不合身的场景，是的，会有更多定制方案**；但他同时预期 Lovable 会**继续与既有工具互操作**——引导用户接 Google Suite / Microsoft / Slack / Salesforce / HubSpot，关键是让用户看清**数据流向**（别把错的数据给错的人）。定位是"底层继续用你喜欢的工具，**上面套一层定制界面**"。

## 呼应与分歧

- ⚠️ **与 [swyx 对模型路由的反对](../videos/20260710-latent-space-swyx-agent-labs.md)正面冲突**：swyx 认为"以路由为傲 = 永远无法榨干任何一个模型 = 最小公分母陷阱"，Osika 则把多模型路由 + 自研窄模型当作核心策略。这是本库应用层战略上最清晰的一处分歧，见 [AI 的商业化与价值捕获](../topics/ai-business-and-value-capture.md)。
- 与 [ElevenLabs 的 Mati](mati-staniszewski.md)、[Legora 的 Max](legora-max.md) 构成"**应用层三件套**"的第三个样本：垂直化 + model-agnostic + 自研窄模型/post-training。三家都是前沿实验室的八九位数客户，同时都在建自己的模型。
- "工程不再是瓶颈 → 决定造什么才是"呼应 [志鹏"编程在编两样东西"](zhipeng.md)（实体 code 被取代、心智模型/设计决策未被取代），见 [AI 与就业](../topics/ai-and-jobs.md)。
- "每次犯错进 agentic 系统改进"是 [评估与基准](../topics/evaluation-and-benchmarks.md) 里"生产数据驱动的私有 eval 循环"的实例；与 [Decagon 的"成熟用例才走开源小模型"](../videos/20260711-all-in-ipo-token-roi-china-open-source.md)是同一逻辑的两种表述。
- "有主见的平台包掉支付/安全/部署"与 [Steven Sinofsky 的"企业里所有有趣的事都是例外处理"](a16z.md)构成张力：Osika 靠**减少例外**取胜，Sinofsky 认为例外正是企业软件不可 vibe code 的原因。
