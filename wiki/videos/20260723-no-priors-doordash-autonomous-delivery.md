## 概要

DoorDash 把自己看成一个**本地商业的操作系统 / 生态**，而不只是外卖网络。两位创始人反复强调同一个方法论——**一切都从实验开始、从用例往回倒推**（这也是 2013 年 paloalto.com 八份 PDF 菜单 + 一个 Google Voice 号码起家的故事）：先搞清楚要解决的客户问题，再倒推该建什么技术，而不是"先把技术造出来再找问题"。这条方法论既解释了 Ask DoorDash，也解释了他们为什么最终决定自研 Dot。

## 核心观点

### Agentic commerce：Ask DoorDash 真的改变了消费行为

- **起点是"对话式"而非"语音"**：两年前他们最初押注**语音**作为交互模态，"结果那个没成"；真正 landing 的是**自然对话体验**——用户能把脑子里的东西直接翻译成界面，不用先做搜索、再做关键词优化（00:00–00:01）。
- **可量化的行为改变**（本节最硬的一手数据）：
  - 餐厅侧：用 Ask DoorDash 的 trajectory 里 **50% 是从没点过的店下单**——而"让用户从新店下单"历来是 DoorDash 最难撬动的指标之一（00:01–00:02）。
  - 杂货侧：**客单量大约高 40%**——用户会拍一张冰箱照片说"帮我补货"、做带饮食限制的 meal planning、或"这周末想给家人做意面"（00:02）。
- Sarah 的反应很关键："我从没觉得 DoorDash 难用，但这说明**存在没被满足的潜在需求**——不是难用，而是'去新地方吃'这件事此前不够容易。"Andy 确认：人会建立习惯，但也想要多样性，对话式体验成了表达这种多样性的自然出口（00:02–00:03）。
- **把 world knowledge 接进来**：模型知识截止太早，所以他们特意接入"DoorDash 之外正在发生的事"——网上在流行什么、论坛里在聊什么——来提升可信度、贴合"想吃点酷的"这种社交货币式需求（00:03）。
- **agent-first 的未来**：Andy 反复提到一个统计——"**现在网上的 agent 流量已经超过人类流量**"；如果今天在车库里重做一个 DoorDash，"大概会是 agentic-first 的样子"（00:04）。上周他们上线了 **DoorDash CLI**，就是在降低 agent 参与这套体验的摩擦。一个具体用例：办公室经理把摄像头对准茶水间货架，货架快空时自动 fire 一个 query 让 DoorDash 补货（"这是个'人类存在任务'"）（00:05、00:47–00:48）。
- **对未来购买方式的判断**：Google 搜索 query 在变长 → 人越来越愿意像对真人一样对 app / agent 说话；把这种摩擦降下来会直接提高下单意愿，且会外溢到食品之外的杂货、便利店、零售（00:46–00:48）。

### 机器人/自动化：一段被低估的八年长跑

- **2018 年就开始，比外界以为的早得多**——当时自动驾驶/机器人还不明显会成。作为 founder-led 公司，他们能"想更远、更投机的事，并持续琢磨怎么不被下一个 DoorDash 颠覆"。而下一个 DoorDash"不会是把同样的东西配个更好的 UI 重做一遍——那太蠢了"，而会是"把 AI agent commerce、自动化、机器人、无人机原生地纳进来的那种"（00:07–00:08）。
- **先做平台、后被迫自研**——三条关键教训（00:09–00:15）：
  1. 头几年他们**不打算自己造机器人**，而是当平台：建 API、管分发，跟"从人行道机器人到 robotaxi"的所有玩家都合作过。这**验证了自动化是"何时"而非"是否"**的问题。
  2. 学会了"**让自动化跑起来要在它周围建一大堆东西**"——基础设施、生态、调度、运营。"不是把一个机器人 plop 进去、或者把一个 LLM plop 进去就自动发生了。"由此建了 **autonomous delivery platform**（后自动化世界里需要的全套 API、dispatch、商户/消费者集成）。
  3. **最重要的一条：必须朝用例造，而不是先造技术再找问题**。"软件世界 YC 教你 serve the customer、build something people want；可一到硬件/hard tech/机器人，大家偏偏反过来先造技术。"市面上的自动化创业公司"都是在真空里造的东西"，没有一个正好是 DoorDash 需要的。
- **为什么现有形态都不对**——DoorDash 的平均配送 **3–5 英里**、典型配送时间约 15 分钟（不含做餐）：
  - **人行道机器人**（2–3 mph 的"轮子上的饮水机"）：技术简单有效，但**速度和距离是硬伤**，"放个 2 mph 的机器人永远不可能 work"。
  - **robotaxi**（4000 磅、为载人设计）："只运几个墨西哥卷，真需要一辆带座椅和空调的 4000 磅车吗？" 而且 robotaxi 的**上下车问题不一样**——Waymo 把你放在半个街区外没关系（你能走），但**包裹不能走**。
- **由此定义出 Dot 的形态**：处在两者之间——"正确的隐喻是一辆**自动的摩托/踏板车/自行车 profile 的车**"：不用 4000 磅、大约 **300 磅**，但要比人行道机器人快得多，跑 **20–25 mph**（00:14–00:15）。没人在造这个，于是他们决定"掌握自己的命运"，自研。

### DoorDash Dot：设计与现状

- Dot 是**完全自研的自动配送机器人**：约 **300 磅**、最高 **20 mph**、体积是汽车的 **1/10**；**唯一一款设计成不只走人行道、还能上自行车道和马路的**配送机器人（00:21）。
- **在 Phoenix（Tempe）已做配送近两年**，去年升级到**全自动 L4**——"到 Tempe 会觉得像在旧金山看 Waymo"（00:21、00:35）。
- **不能直接抄 Waymo**："这是一辆自行车道 profile、在马路和人行道之间不断切换的车，据我所知世界上没有别的东西像它、也没有别的东西表现得像 Dot"——因为是**为配送用例专门造的**（00:36）。

### "做自动化的生意需要的远不止自动化"——规模化的真实边缘案例

这是本期最有价值的一段一手经验（00:27–00:38），把"demo vs 规模化服务"的鸿沟讲得极具体：

- **一次做七天、一天十小时、几百台车之后，东西开始坏**：
  - 传感器被**灰尘**盖住、地上有**落叶**——Dot 靠右像自行车一样走，可能右两轮压在落叶上、左两轮还在沥青上，**要给轮子的扭矩完全不同**，底层控制/中间件必须专门处理，"在漂亮的 demo 环境里我永远想不到这个"。
  - **再生制动的电冲击**：某些场景要急刹到 regen braking 反过来冲击电池，属于极端边缘案例但必须处理，因为安全至上。
  - **开机时间**：demo 期没人在意；原始版本是一个工程师几小时糊的 Jenkins 脚本，单台没问题，但**每天早上几百台一起开机**、脚本一半时间崩、要 30–45 分钟，乘以 500 台就成了巨大的生产力问题。
  - 还有制造/供应链/可靠性、depot 与维护、电池充电、**如何找到客户家门口的确切位置**——Google Maps 的 GPS pin 在公寓楼从来不准，人类 Dasher 能"看一眼就找到那栋楼、那扇门、那道闸门"，机器人不能。
- **这里 DoorDash 的数据优势最锋利**：**"人类 Dasher 历史上把包裹放在哪里"这份 drop-off 数据只存在于 DoorDash，Google Maps 里没有**——正是"最后 100 英尺"问题的解（00:31–00:32）。

### 多模态车队策略：不是全有或全无

- **不为万物设计一台机器**，而是多模态：Dot 做 3–5 英里的密郊配送（从 strip mall 出发，Phoenix 这种密郊是 Dot 的完美市场）；路况差的乡村且是轻单可能用**无人机**；要爬楼梯、pick-and-pack 的复杂杂货单**仍然用人**（00:23–00:24）。
- 对商户和消费者是**同一套体验**：消费者还是同一个 app，商户只需一次集成——"你已经接了 DoorDash，突然之间就同时拿到了 Dasher、无人机、自动化"（00:24–00:25）。
- Sarah 点出一个真优势：**DoorDash 掌握路由和分发**，可以自己决定哪一单交给哪种模态。

### 为什么招得到机器人/AI 人才

- Stanley 的 pitch 很简单："你是想做原型和 demo、待在 PhD lab，还是想做**能真正在现实世界里 ship 的东西**？"（00:26）
- 过去十年 autonomy 领域很多人"做了十年却从没看到产品被真正用起来"；DoorDash Labs 走的是**务实、可落地**的路线——"不是去做疯狂的登月想法，而是把能 ship 的东西放到现实世界里、开始学这些技术怎么跟物理世界交互"。
- 另一张牌是 **world-class 运营团队**：AI 研究者往往不愿碰运营、或以为运营很简单；DoorDash 能让研究者与运营团队搭档去采集/标注数据、部署机器人、跑车队运营——"我们不是在谈假设，你就在 Phoenix 做真实配送"（00:34–00:35）。

### 分布式环境与"猫在洗碗机里"——机器人数据的一手论证

- Stanley：机器人真正难的不是 cherry-picked 的一次成功 demo，而是**让它在任意物体、任意环境里都 work**；最好的真实数据就是**真实世界数据**（00:22–00:23）。
- Sarah 引她投资的 Sunday（家务机器人）呼应："你没法想象真实分布——去做洗碗，结果**一只猫在洗碗机里**（'这猫喜欢洗碗机'）；这不是工程师坐着能想象出来的场景"；就像 Dot 那个"一侧轮子在落叶上的扭矩问题"，"只有数据足够多才知道这类情况有多少"（00:33–00:34）。
- Andy 补充：物理世界"你只能仿真那么多"，接下来 AI 最令人兴奋的前沿是它如何影响物理世界。他们几个月前上线了 **Tasks** 产品——让 Dash 车队的人帮忙采集数据点、训练 world model；坦言这块"非常早期"、form factor 与模型路线都还有分歧（00:19–00:20）。

### 内部 AI enablement：Metis、Dashbench 与失控的支出

这是本库关于**大型非 AI-native 公司如何把 AI 用起来**最具体的一份一手材料（00:39–00:44）：

- **为什么收购 Metis**（去年）：为一个上万人的老公司**注入 AI-native 思维**——很多员工"因为太习惯历史上的做法而看不到什么是可能的"，需要引入见过前沿的人。
- **Dashbench**（几周前发布）：主要衡量各家模型和 harness 在**编码任务**上的表现，"帮我们算清在这上面花的钱的 ROI"。
- **支出曲线**（可核验的一手数字）：**6 月的花费比 1 月涨了约 20 倍**（"20x，不是 20%"）。被问到之后是否降下来，Andy 说**已经 flatline（压平）**——靠有意的治理：年初/去年底大家是在"实验、放手让人跑"，现在到了"一算 ROI、砍掉浪费"的阶段。
- **模型路由的做法**：如果能"把智能最大化、同时把便宜任务下放给开放权重模型"，就能拿到 **Fable 级别的智能却付更少钱**（00:41–00:42）。这与本库 [评估与基准](../topics/evaluation-and-benchmarks.md)、[token 治理](../topics/using-llms-in-practice.md) 里转述的"DoorDash 用内部基准确认开放权重不降质量再路由"完全对上，此处是创始人本人口径。
- **增长最快的是非技术岗**：绝大部分支出仍在工程，但**席位增长最快的是分析师、运营、客户经理**（如自动化和战略商户的 QBR）。
- **一个诚实的能力边界**：他们和前沿实验室合作跑会计/分析任务时发现——**团队随口说"还行"，但真送企业数据去（要做数据脱敏、搭 RL 环境）时模型才 crush，而带全部真实企业数据时又表现不如预期**（00:43–00:44）。呼应 Sunday 机器人的类比："**把问题 dumb down 后模型表现好，但一上真实企业数据就不行**"。悬而未决的问题是：这是 harness 没调好，还是模型的数据分布/能力本身就缺这一块、从而无法在会计/分析/财务上带来 step-function。

### 劳动力预测：10 年后 Dasher 会更多，不是更少

- 被问"终局是不是干掉所有 Dasher、全是 Dot"，Stanley 的预测反直觉：**10 年后会有更多 Dasher 做配送，不是更少**（00:44–00:46）。
- 论证：DoorDash 增长约 **25% YoY**、9 百万 Dasher；再 5–10 倍下去，"总不能让半个美国都来送单"——所以**必须引入新模态 + 提升效率**。未来是一支 **multimodal fleet**（Dot / 无人机 / Waymo / 人行道机器人都要），"每一种模态我们都想拿到手";同时随着配送越来越便宜，需求还会更强地上涨——**"你会看到更多自动化和机器人，但也会看到更多人。"**

## 与本库其它观点的关系

- **对"物理 AI 里就业叙事是反的"的直接印证**：[Applied Intuition](20260721-a16z-applied-intuition-physical-ai.md) 说"没人愿意做的岗位没人接，运营方求着要自动化，那些人现在去跑 DoorDash 和 Uber 了"——DoorDash 创始人从**平台另一端**给出互补数据：不是替代人，而是**在 25% 复合增长下人力供给追不上，只能多模态并进**。见 [AI 与就业](../topics/ai-and-jobs.md)。
- **"做自动化的生意需要的远不止自动化"**是 [物理 AI 与机器人](../topics/physical-ai-and-robotics.md) 里"物理世界更 messy"论断的又一份一手证据——与 [何小鹏"运动控制被严重低估"](20260528-zhang-xiaojun-he-xiaopeng-robot.md)、[沈宇军"落地暴露的自由度 gap"](20260722-zhang-xiaojun-shen-yujun-lingbo-embodied-native.md)、[Applied Intuition"瓶颈在硬件不在算法"](20260721-a16z-applied-intuition-physical-ai.md) 同族，但 DoorDash 讲的是**规模化运营层**（开机脚本、depot、落叶扭矩）而非模型层。
- **"最后 100 英尺"drop-off 数据 = 只存在于 DoorDash 的护城河**，是 [Mark Cuban"窄数据集是魔法"](20260721-all-in-mark-cuban-ai-bubble.md)、[Legora"数据要全不要 80%"](20260714-all-in-11labs-legora-voice-law.md) 在物流场景的又一版本；也呼应 Sarah 自己点出的"很多人对'incumbent 数据优势'的理解太表面——关键是这份数据和你要 agent 完成的用例有没有关系"。见 [AI 商业化与价值捕获](../topics/ai-business-and-value-capture.md)。
- **Dot 的"介于人行道机器人与 robotaxi 之间"** 与 [Kay Ke（PI）"机器人更像大模型而非自动驾驶"](20260716-zhang-xiaojun-kay-ke-physical-intelligence.md)、[Travis Kalanick"按行业定制的专用机器人"](20260722-a16z-travis-kalanick-atoms.md) 一起，构成"通用 vs 专用"光谱上的又一坐标——DoorDash 是**为单一配送用例专门定义形态**的极端。
- **Dashbench / 20 倍支出后压平 / 开放权重路由** 把本库 [token 治理](../topics/using-llms-in-practice.md)（RAMP 21 倍、Chamath 45 天翻倍）里的"组织行为学"补成了当事人口径；"企业真实数据上模型表现打折"呼应 [评估与基准](../topics/evaluation-and-benchmarks.md) 里"dumb down 就好、上真实数据就崩"的反复出现。
- **agent 流量已超人类流量 + agent-first 重做 DoorDash + DoorDash CLI** 与 [Steinberger"每个 app 都是很慢的 API"](20260212-lex-openclaw-steinberger.md)、[戴雨森 agent marketplace](../people/dai-yusen.md) 的 agentic commerce 判断跨场景呼应。

- 平均配送 **3–5 英里**、典型时间约 **15 分钟**（不含做餐）（**断言**，00:13）。

- 预测：**10 年后 Dasher 会更多而非更少**，配送走向 multimodal fleet（**预测**，00:44–00:46）。
