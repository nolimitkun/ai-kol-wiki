# Software in the Age of Agents：headless 软件、护城河与例外处理

## 概要

从 Salesforce "headless 360" 发布切入，讨论 **agent 时代的企业软件**：当 agent 而非人来访问系统，UI 是否还重要？软件的护城河（stickiness）来自哪里？为什么"vibe code 一个 SAP"是严重低估？Sinofsky 以四十年企业软件经验反复强调一个反直觉主题——**企业里所有有趣的事都是例外处理（exception handling）**，而例外没写在任何字段里、只在人的脑子里；以及**生产力永远创造新工作**，长尾不会变短、只会以新形态变长。这是对 [Benedict Evans](20260608-a16z-benedict-evans-ai-economics.md) 宏观"价值上移"命题的**微观补充**：价值确实在上层，但上层是难以复制的组织逻辑，不是模型。

## 核心观点

### headless 软件与 agent 的三类动作
- headless = 承认软件价值不在"给人用的 workflow/UI"，而在**底层的数据与业务逻辑**；agent 通过 API 而非 UI 访问（Salesforce headless 360 更像营销 rebrand，"没有真正改变什么"，但方向是对的）（00:02–00:04）。
- Sinofsky 的实用三分法——agent 到底在做什么：**查（lookup）** 最轻、所有系统都擅长；**改（do something）** 立刻撞上企业软件难题（要冒充特定人、要凭证、算不算一个付费席位）；**析（analyze）** 最适合 agent（不受时间约束、可多模型比对），但也是**幻觉的重灾区**，每步都要能验证（00:07–00:09）。

### 软件护城河到底是什么
- **"最黏的软件就是正在被用的软件"**；最黏的动作是**收钱**（Stripe 首次真正解决了"跨每个司法辖区/币种/税法收款"这一保险级难题）（00:11–00:16）。
- 护城河往往是**没人设计过的怪东西**：没人开会决定"把日历做成 Outlook 的护城河"，但通用汽车不会为了日历迁移 60 万席位——例外的递归依赖、SOP、肌肉记忆才是黏性（00:13–00:14）。
- **SAP 是终极黏性**："把 SAP 从大型车企抽走，公司就蒸发了"——因为公司把业务规则**编码进了产品**；一个反例是保险业跑着 50–75 年前写的软件，至今在招 COBOL 程序员（00:15–00:17）。

### "vibe code 一个 SAP"是严重低估
- **误解**："有个 Postgres + API 就能替代 SAP" → **绝对错误**。SAP 装几年不是因为集成商慢，而是它被**定制到贴合这家公司实际怎么运转**；数据在哪个库里根本不重要，逻辑才重要（00:17–00:18）。
- 十大车企"都只是造车"，区别全在**内部流程**（造什么车、买多少料、对冲哪种货币、何时上新线）——"这些公司实际上是由一群人盯着 SAP 屏幕运营的"（00:20–00:21）。
- Goldman 银行家名言："你不懂，我们靠 Excel 赚的钱比你们（卖 Excel 的）还多"——同一个 Excel，Goldman 的**应用方式**极度差异化（写 add-in、写代码）；企业软件的定制深度被创业者"疯狂低估"（00:21–00:22）。

### 一切有趣的都是例外，一切自动化都是处理例外
- **"企业里几乎所有有趣的事都是例外。"** 销售不认为默认值对任何账户是对的；例外（不同地区如何回应、权限、政策）**不在 Salesforce 字段里，而在某人脑子里**——这就是 agent 需要的 "context graph"（00:29–00:30）。
- 例外收集缓慢：销售周期长、每个例外不会立刻给数据，要"观察足够多交互"才敢说抓全了 context（00:31–00:32）。Amazon 是范例——用"一律判给客户有利"极端简化客服，再用数据在后台持续优化（00:33–00:34）。

### 生产力永远创造新工作（长尾变长而非变短）
- **"人们忘了生产力会催生新场景。"** 自动化最枯燥的一环后，全新的东西会冒出来：Amazon 免了退货客服，但后台多出一整套"如何让它不再发生"的分析（00:35–00:37）。
- **放射科案例**（半正例）："放射科医生都爱 AI，现在我们反而闹放射科短缺"——创新与需求都不是静态的（00:38–00:39）。合同不会因 AI 变少，只会更长更复杂、覆盖更多情景，催生更多诉讼生态（00:37–00:38）。
- 反对"n 个人 + m 份软件的固定馅饼"心态——"AI 的所有负面情绪都来自把待做的工作当成固定的东西"（00:37–00:38）。呼应 [Benedict Evans](20260608-a16z-benedict-evans-ai-economics.md) 的"任务变了但工作没变"。

### MCP / middleware：工程师的一厢情愿
- MCP 之类"一切皆干净 API"是**工程师视角**，与物理现实脱节："没有软件愿意被上面一层善意地抽象成哑数据库"——那是衰败的生意，所以 workday 等**故意不完整暴露 API**（00:45–00:49）。
- 中间件层"在 OSI 网络图里很好看，但从不稳定"；客户也不想从一堆供应商拼装场景（系统只和最不稳的那块一样稳）（00:47–00:48）。

### 创业机会：瞄准两个巨头之间的缝
- 最蠢的是**正面同类竞争**；巨头在技术剧变期唯一不会做的就是扰动自己的现有产品线——他们只会"把 AI 螺栓拧到侧面硬扛过去"（00:53–00:54）。
- 机会在**两个既有玩家之间的缝**、或**组织内两个此前无法对话的职能之间的翻译层**（类比 HTTP/HTML 之于 client-server：没做 client-server 的任何事，却用全新方式实现了那个概念）（00:54–00:56）。
- 企业软件**唯一真正的网络效应在公司内部**：chat 正让"高动机的人"像当年用 Excel 那样在团队里引爆病毒式扩散（00:56–00:59）。
